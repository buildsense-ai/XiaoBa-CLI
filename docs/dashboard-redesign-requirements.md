# CatsCo Dashboard 重设计需求文档

更新时间：2026-05-31

## 一句话目标

Dashboard 不是“功能陈列柜”，而是当前这台机器上的 Agent Body 控制台。普通用户打开它，只需要知道三件事：这个 Agent 能不能运行、缺什么配置、去哪里使用它。

## 设计背景

当前 Dashboard 在长期迭代后混合了启动页、模型页、CatsCo 连接页、Service Manager、Runtime Profile、GauzMem、SkillHub、微信和飞书调试入口。很多区域显示的是开发状态或内部诊断，而不是用户下一步需要做的动作。

新的产品方向是“每个 CatsCo/XiaoBa 进程就是一个独立机器人”。这个机器人可以运行在本地电脑，也可以运行在云端虚拟机。Dashboard 的职责是管理这个机器人所在的运行环境，而不是把所有平台能力都摊开给用户。

## 用户心智

用户不关心 `.env`、connector、bodyId、readiness section、Runtime Profile 这些内部概念。用户关心的是：

- 我有没有登录 CatsCo？
- 这个 Agent 绑定到哪个 CatsCo 机器人？
- 它有没有可用的大模型？
- 它现在有没有启动？
- 我能不能从 WebApp / 微信 / 飞书和它说话？
- 如果出问题，我应该按哪个按钮修？

## 核心对象

- CatsCo 账号：当前打开 Dashboard 的人。
- Agent：CatsCo WebApp 里能被聊天、加好友、进群的机器人身份。
- Agent Body：当前运行 Agent 的机器环境，可以是本机、云 VM 或容器。
- 模型来源：Agent 回复问题时使用的大模型，可以是 CatsCo 托管模型，也可以是用户填写的第三方兼容模型。
- 通道：用户和 Agent 说话的入口，包括 CatsCo WebApp、微信、飞书等。
- 高级诊断：只有出问题、开发或调试时才需要看的内部状态。

## 主要用户路径

### 路径 1：第一次打开 Dashboard

1. 打开 Dashboard。
2. 登录 CatsCo。
3. 选择或创建一个 Agent。
4. 选择模型来源。
5. 启动 Agent Body。
6. 打开 CatsCo WebApp 或本地 Chat 测试。

普通用户应该一直停留在一个主工作台里完成这些动作，不应该被带到多个独立页面。

### 路径 2：已经配置过，再次打开

1. 打开 Dashboard。
2. 顶部直接看到 Agent 名称、运行状态、模型状态和主要入口。
3. 如果 Agent 已经可用，主按钮是“打开 WebApp”或“开始本地测试”。
4. 如果缺配置，主按钮直接指向缺的那一步。

### 路径 3：配置模型

用户不需要进入一个完整“模型页面”。模型配置应该是一个弹窗或抽屉：

- 选项 A：CatsCo 托管模型。
- 选项 B：第三方兼容模型。

第三方模型只需要显示必要字段：

- 兼容类型：OpenAI-compatible / Anthropic-compatible。
- 模型地址。
- 模型名称。
- API Key。

不应该在普通配置入口显示一堆状态解释、readiness 文本、`.env` 说明、运行时细节。

### 路径 4：配置通道

通道不是“启动服务列表”。用户看到的应该是：

- CatsCo WebApp：可用 / 待启动 / 待绑定。
- 微信：选择当前 Agent 后扫码绑定。
- 飞书：选择当前 Agent 后绑定。

每个通道只有两类动作：

- 连接或打开。
- 高级诊断。

微信和飞书属于“当前 Agent 的通信通道”，不是独立机器人入口。

### 路径 5：开发和高级调试

高级能力要存在，但不能占据主流程：

- Runtime Profile。
- 本地 `.env`。
- Connector 进程诊断。
- GauzMem 实验入口。
- SkillHub 开发者功能。
- 原始日志。

这些可以放到“高级”区域，默认折叠，或作为二级导航。

## 推荐的信息架构

### 主导航

主导航最多保留三个普通用户入口：

1. 工作台：启动、状态、模型、Agent 绑定、主要使用入口。
2. 通道：WebApp、微信、飞书等通信入口。
3. 高级：诊断、Runtime、环境变量、实验功能。

如果还需要 Skills，可以作为“能力”入口，但不要和启动配置混在一起。

### 工作台页面

工作台是默认首页。它应该包含：

- 当前 Agent：名称、绑定状态、运行身体。
- 主状态：可用 / 缺模型 / 未登录 / 未绑定 Agent / 未启动。
- 主按钮：下一步唯一动作。
- 次级入口：打开 WebApp、本地测试、配置模型、切换 Agent。
- 折叠诊断：只在需要时展开。

### 模型配置弹窗

弹窗结构：

- 标题：选择 AI 模型。
- 两张选择卡：
  - CatsCo 托管模型：如果还未接入，显示“即将支持”或 disabled；接入后可选择套餐/模型。
  - 第三方模型：表单直接可填。
- 底部按钮：保存。

不单独保留“模型”主页面。

### 通道页面

通道页面用卡片表达：

- CatsCo WebApp。
- 微信。
- 飞书。
- API 或后续通道。

每张卡片显示：

- 是否可用。
- 绑定到哪个 Agent。
- 一个主动作。
- 一个高级诊断入口。

### 高级页面

高级页面可以保留历史功能，但要明确这是调试区：

- 运行诊断。
- Connector 进程。
- Runtime Profile。
- 本地环境变量。
- GauzMem beta。
- SkillHub beta / Developer。

## 非目标

这次重设计先不解决：

- 组织权限模型。
- 完整多用户审计。
- GauzMem 图记忆隔离策略。
- 跨设备文件 grant 的最终安全模型。
- CatsCo 托管模型后端接入。

但界面要预留这些未来能力的位置，不能让现在的临时设计堵住后续路线。

## 必须删减的东西

普通用户主流程里不应该直接出现：

- 大面积 readiness 卡片。
- `.env` 表格。
- 复杂模型状态说明。
- ServiceManager 原始服务列表。
- “配置 / 模型 / 服务”这种内部分类。
- 过多解释性长文案。
- 多个看起来都像主按钮的操作。

## 视觉原则

- 主屏首屏只回答一个问题：现在能不能用，不能用下一步按哪里。
- 少用大段说明文字，尽量用状态、按钮、短标签表达。
- 卡片用于“对象”，不要把页面区块都做成卡片堆叠。
- 普通状态不需要过度展示；只有异常状态需要具体解释。
- 主按钮每屏最多一个。
- 高级项默认收纳。
- 移动端优先保证操作清晰，不追求把所有状态展示完整。

## 工程约束

- 现有 Dashboard 是单文件 `dashboard/index.html`，短期可以继续在这个文件内改，但要避免继续堆叠页面。
- 模型保存接口已有 `/api/settings`，第三方模型配置可以复用。
- CatsCo 登录、Agent 绑定、body 状态、connector 状态已有 API 基础，但部分状态仍需要兼容旧服务端。
- 不能破坏现有本地 agent 启动路径。
- 旧 `.env` 仍承载第三方模型、Feishu、Weixin 等配置，直到后续配置模型重构完成。

## 新 Dashboard 的判断标准

一个新用户打开 Dashboard 后，应该在 10 秒内知道：

- 我现在配置到哪一步了。
- 还缺哪一步。
- 应该点哪个按钮。
- 配完后在哪里聊天。

一个老用户打开 Dashboard 后，应该在 5 秒内完成：

- 查看 Agent 是否在线。
- 打开 WebApp。
- 或发现并处理阻塞项。

一个开发者打开 Dashboard 后，仍然能找到：

- 原始运行状态。
- connector 诊断。
- Runtime 和 `.env`。
- 但这些不干扰普通用户。

## 需要 subagent 独立评审的问题

1. 如果完全不沿用现有页面，Dashboard 最少需要几个主入口？
2. 模型配置应放在弹窗、抽屉还是工作台内联区域？
3. 微信/飞书通道应该放在工作台首屏，还是单独通道页？
4. 高级诊断如何保留但不污染普通流程？
5. 现有 “启动向导” 是否应该改成“Agent 工作台”？
6. 第一版实现应该大改 DOM，还是先做外观和信息架构重排？
