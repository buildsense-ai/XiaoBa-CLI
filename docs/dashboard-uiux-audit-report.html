CatsCo Dashboard · UIUX Audit · 2026-05-31

把 Dashboard 从“开发调试台”改成“启动 Agent 的工作台”

当前版本已经比之前清爽，但交互重心仍然不够明确：用户第一次打开时，右侧 Chat 占据最大面积，左侧启动流程却是窄列； 视觉上有登录、诊断、状态、WebApp、模型、微信等多个同级入口，容易让人不知道“我现在应该先点哪一个”。

一、审计结论

主问题

页面仍然在展示系统，而不是引导用户

主屏最大区域是不可用的黑色 Chat；真正需要处理的登录、模型、绑定、启动被挤在窄侧栏。

视觉问题

红色和状态标签过多

“login / 待配置 / setup / needs attention”等状态在无错误时也像报警，造成焦虑感。

可取方向

三步启动模型已经成型

启动、模型、通道这三个主入口是合理的，下一步应把它们做成一个连续向导，而不是分散页面。

二、当前截图与问题标注

启动页 · 桌面

  • 信息层级倒置：Chat 占了约 65% 宽度，但未登录时不能用；启动流程只占窄列。
  • 登录入口重复：有红色登录状态卡、登录或注册按钮、下面又有完整登录表单。
  • “打开 WebApp”同时出现在顶部和侧栏底部，视觉权重重复。
  • “login”英文标签、`needs attention` 英文标签破坏中文产品感。

启动页 · 窄屏

  • 左侧 icon rail 仍然占宽度，但用户看不懂 @、~、>、# 分别是什么。
  • 第一屏出现两套登录 UI，用户会疑惑“点上面的登录还是填下面的表单”。
  • Chat 卡片露出一截但不可用，造成页面像没加载完。
  • 移动端更适合单列向导：先登录，再模型，再启动，再打开聊天。

模型页 · 桌面

  • “先完成关键配置”作为概览有价值，但它又重复展示了模型、CatsCo Chat、Runtime Profile 三张状态卡。
  • 当前版本实际只需要用户完成自定义模型，所以页面可以更直接：选择模型来源、填 key、测试连接、保存。
  • Runtime Profile 是高级能力，已折叠是对的，但标题仍在主视线内，普通用户可能误以为必须理解。
  • 蓝色提示条、橙色标签、红色待配置同时出现，颜色语义不够克制。

通道页 · 桌面

  • “CatsCo agent / 飞书机器人 / 微信机器人”更像服务管理，不像用户视角的入口管理。
  • “启动微信”不等于“微信扫码绑定当前 agent”，主动作的含义不够贴近目标体验。
  • 日志按钮在普通用户界面权重偏高，应进入每张卡的更多菜单或诊断区。
  • 页面大量留白是好事，但卡片内容仍然偏工程词：connector、agent、本地 agent。

通道诊断 · 展开

  • Version、Runtime、Model、Skills Path 这类信息对排障有用，但不应该和通道配置同级。
  • 路径长文本在卡片里很重，建议改成复制按钮和一行摘要。
  • 诊断区可以保留，但视觉应该明显低于用户操作卡片。

三、信息架构建议

建议把 Dashboard 定义成 body console，但默认只暴露“启动当前 agent”的任务流。 用户不是来理解 body/status/readiness 的；用户是来让这个 agent 能工作，然后去 CatsCo WebApp、微信、飞书或本地 Chat 使用。
当前页面 保留 降级/隐藏 建议改名
启动 登录、模型状态、agent 绑定、启动 connector、打开 WebApp 不可用 Chat、重复登录表单、body 细节、英文状态 首页 / 启动 Agent
模型 托管模型 / 自定义模型选择、自定义模型表单、测试连接 Runtime Profile、env、readiness 细节 模型与额度
通道 CatsCo WebApp、微信、飞书、API 入口 PID、uptime、日志按钮、服务命名 聊天入口
实验/开发 GauzMem、Skills、开发者中心、伙伴 默认不在主导航打扰普通用户 实验室

四、视觉设计建议

布局

未就绪时不要把 Chat 作为主画面

未登录/未配置模型时，主区域应该是 3-4 步 setup checklist；Chat 只在 ready 后成为主区域。

颜色

红色只用于真正错误

未登录、未配置属于正常初始化状态，建议用蓝/灰；红色只给 token 无效、body 冲突、启动失败。

字体

统一中文产品语言

把 login、setup、needs attention、readiness、Diagnostics 等改成中文，英文只留给开发诊断展开区。

组件

减少卡片套卡片

当前模型页和启动页都存在卡片内再放卡片的感觉。建议用步骤列表、分段表单和单个主 CTA 替代。

五、建议的大改方向

方案 A：启动向导首页

第一屏显示 4 步：登录 CatsCo、配置模型、选择 Agent、启动。每步只给一个主动作。
Ready 后首页变成“Agent 正在运行”，主 CTA 是打开 WebApp，次 CTA 是微信扫码、本地 Chat、通道管理。
Chat从首页主区域移到独立标签，或只在 ready 后展开。

方案 B：工作台首页

顶部Agent 名、运行状态、当前账号、主 CTA。
中部三个大块：使用入口、模型、通道。
底部高级诊断、日志、Runtime Profile、实验功能。

六、子 agent 审视重点

审视角色 要回答的问题 希望输出
产品/信息架构 首次用户是否知道下一步？哪些入口可以删除或合并？ 主流程、页面分组、命名建议。
视觉/UI 颜色、字体、卡片、留白、状态标签是否专业？ 视觉优先级、组件风格、具体可改点。
前端实现/响应式 哪些调整能快速落地，哪些会牵扯后端或状态管理？ 低风险改动顺序、测试点、潜在风险。
