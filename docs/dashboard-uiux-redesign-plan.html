CatsCo Dashboard · Redesign Proposal · 2026-05-31

新的交互原则：先启动 Agent，再进入 Chat

三个独立 reviewer 的意见高度一致：Dashboard 当前最大问题是“未就绪时把 Chat 当主界面”。改版应把首页改成状态驱动的启动向导； Chat、日志、Runtime Profile、GauzMem、Skills 等都不应该在普通用户的第一路径里抢注意力。

一、三方 review 共识

P0

未就绪时 Chat 不能占主画面

当前最大黑色区域不可用，却占据最强视觉权重。首页应先回答“还差哪一步才能让 Agent 工作”。

P1

状态语言像报警，不像进度

未登录、未配置属于初始化，不是错误。红色只应留给 token 失效、body 冲突、启动失败。

P1

主入口要变成用户语言

“connector / readiness / Runtime Profile / Diagnostics”应退到高级区；主入口应叫“网页聊天、微信、飞书、本地测试”。

二、当前界面的根因

现状：启动流程被压缩，Chat 抢主视觉

用户看到的是一个不可用的聊天窗口，而不是清楚的 setup 流程。

移动端：icon rail 和露出的 Chat 都干扰任务

小屏应该只显示当前任务和下一步，不能保留桌面式侧栏。

三、目标信息架构

Dashboard 的默认用户任务不是聊天，而是让这个 body 上的 agent 能工作。 所以首页是“启动 Agent”，Chat 是 ready 后的使用入口或本地测试入口。
新导航 用户理解 包含内容 移除/降级
首页 我还差哪一步才能启动这个 Agent 登录、模型、Agent 绑定、启动入口、ready 后打开 WebApp 不可用 Chat、重复登录表单、英文状态
模型与额度 这个 Agent 用什么模型 托管模型、自定义模型、访问凭证、保存，未来接额度 Runtime Profile、CatsCo Chat readiness
聊天入口 我从哪里和 Agent 聊 CatsCo WebApp、微信、飞书、本地测试 Chat、API PID、uptime、connector 命名、日志主按钮
高级 排障和开发 诊断、日志、Runtime Profile、.env、GauzMem、Skills、开发者中心、更新 默认不在主路径里展示

四、首页目标交互

1

登录 CatsCo

先确认当前操作者是谁。未登录时只出现一个主按钮：登录。

2

配置模型

当前阶段优先自定义模型；托管模型显示为即将接入。

3

选择 Agent

绑定这个进程对应的 Agent/body，避免复制 API key 造成多 body 冲突。

4

打开入口

启动后主按钮变成打开 CatsCo WebApp；微信、飞书、本地 Chat 作为次入口。

六、状态和颜色规范

状态 颜色 使用场景 不要用于
未开始 灰色 还没登录、还没配置、还没绑定 不要用红色
待设置 蓝色 当前下一步，例如登录或配置模型 不要同时出现多个蓝色主 CTA
运行中 / 已完成 绿色 connector running、模型配置有效、Agent 已绑定 不要用于“可以尝试”
需要处理 琥珀色 平台状态未知、旧平台未支持、非阻塞提示 不要用于真正失败
错误 红色 token 失效、body 冲突、启动失败、权限错误 不要用于正常初始化

七、实施顺序

Phase UX-1 · 已实现

把首页从 Chat 改成启动向导

  • 默认首页不再显示不可用 Chat。
  • 合并登录卡和登录表单，未登录时只有一个主登录入口。
  • 中文化状态：未登录、待设置、未开始、运行中、错误。
  • 移动端取消 icon rail，改成中文顶部/底部导航。
  • Chat 仅作为 ready 后主界面或“本地测试 Chat”入口。
Phase UX-2 · 通道页已改

模型与 Agent 通道拆清楚

  • 模型页只做模型来源和保存，不再重复 CatsCo/Runtime readiness。
  • 通道页已从 service manager 改成 Agent 通道：网页、微信、飞书只表达是否绑定到当前 Agent。
  • 启动/停止 connector、日志、路径、PID、Runtime Profile、.env、更新全部进入高级。
Phase UX-3 · 需要后端配合

补真实能力

  • 模型测试连接 API。
  • CatsCo 托管模型和额度。
  • WebApp 微信 channel secret 同步到 body。
  • openid / 飞书 user id 绑定回 CatsCo actor。
