核心重构视角

CatsCo 不只是装在电脑里的个人助手，而是组织可以部署和培养的虚拟员工。

对用户来说，它仍然像一个强大的个人专家：直接聊天、独立处理、不阻塞别人。对系统来说，它必须有组织身份、岗位边界、稳定身体、工具链、长期记忆、权限和审计。

个人助手是“服务对象为 1”的特例。真正的目标形态是：一个虚拟员工同时服务整个组织的人，每个人都有独立 session，但能力和世界模型沉淀在同一个 agent 身上。

三个概念的区别

重构最重要的不是“本地还是云端”，而是谁是主体、记忆归谁、凭证归谁、组织能否长期沉淀能力。

产品定义
Personal Assistant

个人助手

主体是个人用户。记忆、偏好和本地资料围绕这个人展开。适合单人效率，但组织无法稳定复用。

Cloud Bot

云端机器人

主体是触发器或任务。适合自动执行，但通常缺少岗位身份、长期经验和多人协作语境。

Virtual Employee

虚拟员工

主体是 agent 自己。它属于组织，有岗位、身体、工具、凭证、记忆和交付标准，可以多人协作调用。

当前代码现实

现有代码已经有多 session、CatsCo Web connector、runtime profile、GauzMem、local file grant 等基础；这个分支已经完成 body 绑定安全、平台联系人入口、TopicContext 和消息身份 metadata 的 beta 第一版。但产品形态仍然更接近“本地单 agent + beta 平台关系”，还不是完整的组织虚拟员工系统。

CatsCompany 平台现状

当前 bot 仍然是 `users(account_type=bot)` + `bot_config.api_key`，但平台分支已经有 active body lease、body status、`/api/agents`、`/api/agents/open`、`topic_contexts` 和消息 metadata 持久化。也就是说，平台已经能让成员找到 agent、打开会话，并让后续消息带服务端生成的身份上下文。

下一步平台深化

下一步不再是 body status，而是把 beta 关系升级成组织关系：谁能看到哪个虚拟员工、谁能邀请成员、谁是 owner/member/viewer、权限快照如何随 topic 保存。当前的 friend/public 只是可用桥接。

目标体验

普通用户不需要理解 agent body、connector、session key。他们只需要像找一个专家同事一样，登录、找到虚拟员工、聊天、授权资料。

目标数据模型

这个模型不是为了增加角色，而是为了把不同层级说清楚：CatsCo actor 是说话的人，VirtualEmployee 是被叫到的虚拟员工，AgentBody 是它运行的进程/设备，ChannelBinding 把微信/飞书等外部 ID 绑定回 CatsCo actor，UserDevice 是这个 actor 名下可授权的设备，ConversationSession 是 actor/topic 和 agent 的独立对话，SessionProvenance 是每轮消息的来源记录，DeviceGrant 是临时资料授权。GauzMem 可以以后再接这份 provenance。

运行时数据流

现在 connector 直接把渠道 ID 转成 session key。目标是从 CatsCo actor 身份出发，把 CatsCo Webapp、微信、飞书等入口统一成 MessageEnvelope，再由 SessionRouter 决定 session，最后把 identity 传进工具、日志和记忆。

当前关键问题

`Metrics`、`.env`、`process.env.CURRENT_AGENT_DISPLAY_NAME`、固定 service name 等仍是进程级共享状态。多用户 session 虽然隔离，但多个虚拟员工不能安全共进程。CatsCompany 已有 beta TopicContext 和消息 metadata，但还没有正式的组织成员、agent access、角色权限和权限快照表。

目标不变量

任何一个 turn 都不能修改会影响另一个 turn 的身份、凭证、metrics、工作目录或记忆 retrieve 上下文的全局状态；身份/provenance 要结构化保存，而不是只塞进提示词。

本地电脑访问不是默认能力

云端虚拟员工可以像“住在你电脑里”一样处理资料，但前提是用户明确授权。更准确的说法是：用户电脑不是 agent body，而是相对 agent body 的“个人云盘/资料盘”。agent 在云端身体里思考、运行工具、沉淀 session；用户设备只通过轻量连接器提供已授权的文件和能力。

权限边界
云盘视角： 把用户电脑变成稳定资料入口，比把完整 agent 跑在用户电脑上简单很多。本地轻量应用只需要登录、心跳、声明能力、列文件、读写已授权范围、上传/下载流、撤销授权；模型调用、复杂工具链、长期记忆、任务处理都留在 agent body。
来源 今天的例子 目标规则
agent body 自有文件系统 云 VM workspace、本地 runtime workingDirectory 常规文件工具可访问，但受 body policy 限制。
聊天附件缓存 CatsCo/微信附件下载到 `tmp/downloads` 或 `files/weixin` 只能按附件 provenance 读取，不能当作用户一般文件库。
用户授权设备路径 用户选择本地文件、授权目录、SSH 主机 必须先落到 UserDevice，再通过 DeviceGrant 工具访问，记录审计，可撤销。
新的融合点： ChannelBinding 只解决“微信/飞书里说话的人是谁”；DeviceRegistry 解决“这个 CatsCo actor 有哪些可授权设备”；DeviceGrant 解决“当前 agent 能在某台设备上访问什么范围”。所以微信里说“去我电脑拿文件”时，agent 应该先查 actor 的设备清单，而不是默认访问 agent body 的文件系统。
不是硬编码意图流程： 不应该写成“用户说去我电脑就固定调用 A/B/C”的规则。更合理的是把边界写清楚：system prompt 告诉 agent “你的 body 文件系统”和“用户设备”是两类来源；工具描述提供 `list_user_devices`、`request_device_grant`、`read_device_file` 等能力；平台和工具层只负责校验 actor、device、grant 是否匹配。语义判断交给 agent，边界执行交给工具。

单独需求：群聊寻址与回复策略

Session identity 第一阶段已经完成；Reply Policy 先只作为独立需求记录，不进入当前代码路径。Session identity 解决“谁说的”；Reply Policy 以后再解决“这条消息是否该启动 agent，以及是否该公开回复”。

Backlog
当前进度（2026-05-27）： 已完成前置依赖：CatsCo 每轮消息带 SessionIdentitySnapshot，prompt/tool/log 可见；busy queue 合并消息会保留每条 actor/topic/seq；Dashboard 登录和 bot 绑定已拆成显式步骤；平台消息现在也有服务端 TopicContext 和 metadata provenance。Reply Policy 代码实现仍暂缓，避免现阶段引入 hard-coded 群聊寻址规则。这个需求保留为后续 agent-to-agent / 群聊协作的单独 backlog。

当前问题

群聊里任何消息都可能被当成 query。agent-to-agent 群聊还可能互相触发回复循环。只靠模型自己决定“不回复”不稳定，因为 ReAct agent 往往用最终 message content 结束循环，而部分模型也不一定稳定使用 `send_text`。

目标边界

在进入 `AgentSession.handleMessage()` 前先做 addressing/reply gate。无关消息不启动完整 agent loop；相关但无需公开回复的消息进入 observation；只有明确需要回复时才允许 final assistant content 或 `send_text` 发出去。

实现原则： 这里不应该变成复杂关键词路由。未来如果要做，也应优先依赖平台 metadata 里的 mention、reply_to、assigned_agent、sender_kind 等硬信号；文本 @ 名称最多作为兼容 fallback。复杂语义可以交给轻量 LLM judge，但不要在当前阶段为了一个未来场景扩大主线改动。
场景 默认决策 触发完整处理的条件
私聊 handle_and_reply 默认认为用户就是在找这个 agent。
群聊普通消息 ignore 或 observe @ 当前 agent、reply_to 当前 agent、显式分配任务、命中 watch topic。
其他 agent 发言 observe 明确 handoff、任务链路指定当前 agent 接手、或者直接提问当前 agent。
相关但无需公开回复 handle_silent 作为 runtime observation 进入 session，但不允许默认 final message 外发。

最容易返工的风险

这些点如果处理顺序错了，会导致看似完成了 UI，底层仍然是单人 assistant 模型。

风险控制
不能贸然先改 session keyrestore、logs、subagent callbacks、busy queue 都依赖旧 key。先 identity sidecar，再双写。
不能让 legacy setup 回退成隐藏绑定Dashboard 已经走显式 create/select/bind；`/api/cats/setup` 只应作为受限兼容入口，后续可以返回 409/410 引导旧调用方迁到显式绑定。
平台不能缺席第一版 active body lease 已完成，但目前仍是 WebSocket 单服务内存锁；多实例部署时还要升级成 Redis/DB 共享 lease，并把 body 状态展示给用户。
微信/飞书不是平行身份外部 channel id 必须绑定回 CatsCo actor。当前 WeChat connector 还持有 `get_updates_buf` 和 `context_token`，身份绑定要先做，transport 所有权后迁。
安全不要喧宾夺主读写工具、shell、send_file 的严格权限可以后置。现在先把 actor/grant/body 字段放进上下文，留下未来控制口。
多虚拟员工不宜先共进程AIService、ToolManager、SkillManager、env、profile 仍可能共享。初期一 agent body 一进程更稳。
GauzMem 后接，不和第一步绑死当前先把 session/turn provenance 做清楚；图记忆以后从这份结构化来源信息里取 actor/session awareness。
