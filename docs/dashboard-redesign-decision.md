# CatsCo Dashboard 重设计决策

更新时间：2026-05-31

## 结论

新版 Dashboard 采用三入口结构：

1. 工作台
2. 通道
3. 高级

不再把“模型”作为主页面，不再把“启动”作为产品入口名称，不再把 Service Manager、readiness、`.env`、Runtime Profile 等诊断内容暴露在普通用户主流程里。

Dashboard 的默认首页是“Agent 工作台”。它只回答一个问题：当前这个 Agent Body 能不能用；如果不能用，下一步点哪里。

## 为什么这样选

三个独立评审视角都收敛到同一个判断：

- 产品视角：用户不是来管理模块，而是来管理当前 Agent。
- 视觉视角：首屏应该只有一个主状态和一个主按钮，减少解释文字和调试状态。
- 工程视角：第一阶段不应全量重写，应保留现有 API 和关键函数，先做信息架构重排和 DOM 收敛。

所以第一版不追求重构所有底层代码，而是先把可见体验改对。

## 采用的设计

### 1. 首页改为 Agent 工作台

旧首页叫“启动”，但用户心智不是启动服务，而是看这个 Agent 是否可用。新版首页应显示：

- 当前 CatsCo 账号
- 当前 Agent
- 当前 Agent Body
- 模型状态
- 通道状态
- 一个唯一主按钮

主按钮由当前状态决定：

- 未登录：登录 CatsCo
- 已登录但缺模型：选择 AI 模型
- 已登录但未绑定 Agent：选择或创建 Agent
- 已绑定但未启动：启动 Agent Body
- 已可用：打开 WebApp
- 异常：修复问题

### 2. 模型配置改成弹层，不再是页面

模型只是启动条件，不是主目的地。

新版模型入口从工作台的主按钮或次级按钮打开。桌面端可以做右侧抽屉，移动端做全屏 sheet。它交互上等价于“弹窗”：不离开工作台，不进入独立模型页。

弹层只保留两种选择：

- CatsCo 托管模型
- 第三方兼容模型

第三方兼容模型只显示必要字段：

- 兼容类型
- 模型地址
- 模型名称
- API Key

Runtime Profile、`.env`、readiness 文本、adapter endpoint 不进入这个弹层。

### 3. 通道页只表达“怎么联系当前 Agent”

通道页不是服务管理器。它只显示：

- CatsCo WebApp
- 微信
- 飞书

每个通道卡片只显示：

- 当前状态
- 绑定到哪个 Agent
- 一个主动作
- 一个高级诊断入口

微信扫码绑定应明确显示“绑定到当前 Agent”，不能复用日志弹窗或让用户误以为绑定的是账号本身。

### 4. 高级页承接历史和开发功能

高级页放：

- readiness 原始检查
- Runtime Profile
- 本地 `.env`
- connector 进程诊断
- logs
- GauzMem beta
- SkillHub / Developer beta
- Companion beta

这些能力可以保留，但默认不干扰普通启动和使用。

## 不采用的设计

### 不继续保留“模型”主导航

原因：模型配置不是独立工作流。把它放在主导航，会让用户以为 Dashboard 是模型配置工具。

### 不把本地 Chat 常驻在未就绪首页右侧

原因：未配置时它只是一个锁住的空框，占用首屏注意力。应改成可用后的“本地测试”入口。

### 不在通道页展示大面积 readiness

原因：通道页是给用户管理沟通入口，不是看系统检查报告。readiness 应进入高级诊断。

### 不把微信作为启动流程必需步骤

原因：微信只是一个通道。Agent 可通过 WebApp、本地测试、飞书等入口使用。微信绑定应放到通道页。

## 第一阶段实施方案

### 目标

在不破坏现有功能的情况下，把主体验改成新版信息架构。

### 改动范围

- 保留现有 `page-chat` 的 DOM id，但展示名改为“工作台”。
- 主导航只保留“工作台 / 通道 / 高级”。
- 删除“模型”主导航入口。
- 新增模型配置弹层，复用 `/api/settings` 保存逻辑。
- 把 Runtime Profile、`.env`、readiness、Diagnostics、connector 诊断移动到“高级”。
- 通道页只保留 WebApp / 微信 / 飞书通道卡片；本地服务控制折叠到高级诊断。
- 首页收敛成一个主状态、一个主按钮、少量状态摘要。

### 保留的底层能力

- CatsCo 登录和注册
- Agent 选择/创建/绑定
- CatsCompany connector 启动
- WebApp 打开
- 微信 token / 扫码绑定
- 自定义模型保存
- Runtime Profile 编辑
- `.env` 兼容编辑
- connector logs

## 第二阶段实施方案

把 Dashboard 状态进一步正规化：

- `buildCatsChatStage()` 成为唯一主状态机。
- 工作台、通道页、侧边按钮都读同一份状态。
- 减少正则 HTML 测试，增加 DOM/视觉测试。
- 模型弹层、Agent 选择、微信绑定抽屉逐步组件化。

## 第三阶段实施方案

产品能力成熟后再做：

- CatsCo 托管模型真正可选。
- 飞书通道正式产品化。
- 旧 `.env` 逐步降级到只读或完全迁移。
- GauzMem 和 Skills 如果成为普通用户价值，再从高级区提升出来。

## 第一版验收标准

- 新用户打开 Dashboard，10 秒内能知道下一步点哪里。
- 已配置用户打开 Dashboard，5 秒内能打开 WebApp 或看到阻塞原因。
- 普通用户首屏看不到 `.env`、Runtime Profile、readiness、connector 原始状态。
- 模型配置不再需要进入“模型页面”。
- 首页同一时间只有一个主按钮。
- 微信和飞书被表达为“当前 Agent 的通道”，不是独立机器人或启动必需步骤。
