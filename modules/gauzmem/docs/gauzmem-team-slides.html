GauzMem：让 agent 不只是搜索，而是会联想。

0.2 已完成 query-driven memory core。0.3 先做 manual / offline dreaming prototype；长期方向是让图在真实请求之外继续生长。

为什么做记忆不是更大的检索框

关键词和 grep 很强；缺的是“应该想到”。

大型项目里，agent 往往能解决眼前问题，却不知道已有模块、旧约束、旧偏好和当前任务有关。

一次典型 recallend-to-end

从用户问题到临时 bundle，0.2 通常先走 graph，再按需 construct。

1. query

例如：我现在更偏好什么语言？

2. graph first

先扫已有 node / edge，披露局部子图。

3. judge

root relevance 选中或拒绝 evidence。

4. construct if needed

图不够时才搜 source。root 冷启动写 standalone node；node-local construct 写 edge。

5. bundle

只把被选中的线索临时注入 agent。

这不是每轮必经的线性流程。Construct 触发条件包括冷启动、graph seed 未选中、没有新增有效 route、当前 selected graph 不足且有 source roots。

0.2 代谢已实现

遗忘按 agent 的认知 tick 发生，不按墙上时间。

信号 当前含义 注意
selected 真实 query 中被 root relevance 选中。 强强化，增加普通 selectedCount。
rejected root relevance 判断无关或误导。 降权，尤其是 edge。
effectiveWeight 运行时根据 weight、selectedCount、lastTouchedTick 计算。 不直接改 numeric weight。
sweep 显式物化 active -> faint 或 faint -> dormant。 写 decayed event，dashboard 可复盘。
0.3 假设planned
Proposal

如果只靠真实 query，联想图可能长得太慢。

0.3 先做手动 / 同步的 dream prototype：从已有 evidence 自我发问，验证能否提前补出有用联想边。

要验证

后续真实 query 是否更快命中相关线索。

要防止

内部梦境自我强化，污染真实 recall。

0.3 dream episodeplanned

Dreaming 仍然做 root relevance；否则就是盲目建图。

1. seed

选 retrievable、non-dormant、低度或最近的 evidence node。

2. question

生成短问题作为 dream root，不把问题当 evidence。

3. retrieve

用 initialNodeIds 激活 seed，bundleTarget=internal。

4. construct

只写 dream-discovered source evidence。

5. state

dream_selected 弱强化，不进 prompt，不增加普通 selectedCount。

0.3 边界planned

第一版 dreaming 要保守：手动、同步、可复盘。

做 暂不做 原因
CLI / API 手动触发 dream。 自动后台 daemon。 先看污染、成本和收益。
从 retrievable non-dormant node 选 seed。 dormant recovery。 复活策略需要单独设计。
dream_created 低初始权重。 从 dream-created node 继续 dream。 避免内部图自我膨胀。
Dreaming 独立 dashboard 区域。 混进 Recall Replay。 区分用户会话和内部 episode。

安全规则：dream-created 是梦境中发现的 source evidence，不是生成摘要；whyRelevant 是联想说明，不是事实声明。

交付拆分first PRs

这些是实现入口，不是新概念；建议按依赖顺序推进。

方向 第一批改动 验收方式
1. Core initialNodeIds、bundleTarget、dream state profile。 initialNodeIds 只激活起点；bundleTarget=internal 不返回 prompt bundle；dream_selected 不增加普通 selectedCount。
2. Dream policy seed picker、generateDreamQuestions、dry-run plan。 同一 store 能输出稳定 seeds/questions；dry-run 不写 nodes/edges/state。
3. API / CLI / UI POST /v1/dream、gauzmem dream、dashboard Dreaming 区域。 dream runs 进独立区域；不混进用户 Recall Replay。
4. Evaluation dream 前后真实 query 对比。 命中率、无关 bundle 率、成本、图增长都有记录。

0.3 成功标准：dream 长出的边，后续真实 query 能命中，并减少 agent 主动搜索成本。
